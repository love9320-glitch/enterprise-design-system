# 디자인 시스템 규칙서

이 프로젝트에서 UI 작업을 할 때 따라야 할 규칙을 작업 유형별로 분리해 정의한다.
**필요한 파일만 읽어** 토큰을 절약하면서 일관성을 유지한다 (progressive disclosure).

## 핵심 원칙 (반드시 준수)

규칙을 **적용 범위별 3구역**으로 나눈다 — (A) 피그마 작도·코드 작성 양쪽에 적용되는 **공통 설계 원칙**, (B) **코드 작성 전용**, (C) **Figma 작도 전용**. 다른 규칙·메모리에서 "규칙 N"으로 참조하는 번호는 아래 통합 번호를 가리킨다.

> **신설 규칙 번호(15~)**: 15=카피 띄어쓰기(A구역, 아래) · 16=목록 버튼그룹 순서(`templates/list-page.md`) · 17=아이콘 전용 셀 width=iconCellWidth(`templates/list-page.md`) · 18=모달 안 무한 스크롤 테이블 설정(`templates/modal.md`) · 19=알파 라인은 outline(B구역, 아래) · 20=키보드 Tab 체인 규약(B구역, 아래). 이후 신설 규칙은 21부터 이어 붙이고 여기 색인에 등록한다. (신설 5개 도달 — 2026-07-14 가지치기 감사 수행: 비대 행 아카이브 2차 이관, 15~19 전부 사용 중이라 통합·삭제 없음)

### A. 공통 설계 원칙 (피그마·코드 양쪽 적용)

1. **컬러는 시멘틱 토큰 경유** — Base 토큰 → Semantic 토큰 → 컴포넌트. 하드코딩 HEX/rgba 금지. (코드: 시멘틱 토큰 클래스 / 피그마: 컬러 변수 경유) **필요한 시멘틱 컬러 토큰이 없으면 임의로 정의·추측하거나 base·다른 시멘틱으로 우회하지 말고, 멈춰서 사용자에게 "이 토큰이 빠졌으니 추가해 달라"고 알린 뒤(토큰명·용도·예상값) 추가되면 다음 스텝을 진행한다(2026-06-26 지시, 상세 foundation.md).**
2. **간격·라운드·스트로크는 등록 토큰만** — `spacing-*`, `round-*`, `border-*`만 사용. Tailwind 기본값(`p-4`)·임의값(`p-[12px]`) 금지. (피그마도 등록된 간격/라운드 변수만 사용)
3. **디자인 시스템 우선 — 페이지 → 템플릿 → 컴포넌트 순으로 적용** — UI를 만들 때 요구/이미지 구성을 먼저 파악하고 **(1) 페이지 → (2) 템플릿(TableTemplate 등 복합) → (3) 개별 컴포넌트** 순서로 우선 매칭한다. 큰 단위(페이지/템플릿)로 커버되면 그것을 쓰고, 안 되는 부분만 하위(컴포넌트)로 내려가 조립한다. 항상 기존 DS 자산을 우선하며 임의 신규 제작은 지양한다. (피그마: 연결된 인스턴스 / 코드: 기존 컴포넌트·템플릿 import. ※ 페이지 단위 규칙은 페이지 체계가 만들어지면 별도 보강)
4. **바로 만들지 말고 분석 → 매핑 → 조합 → 정리 순으로** — 요청을 즉시 만들지 말고: **(1) 요소 분석** — 필요한 요소·상태를 목록화하고, **디자인 폰트(Pretendard) 가용성을 점검**한다(없으면 설치; 설치 불가 환경에서만 텍스트 변경을 보류하고 컴포넌트 기본 텍스트 유지 — 대체 폰트로 바꾸면 자간·정렬이 깨짐). **(2) 매핑(페이지>템플릿>컴포넌트)** — 가장 큰 단위로 덮을 DS 자산 선택(예: 모달+테이블 템플릿으로 대부분 커버). **(3) 조합/배치** — 고른 자산을 그대로 배치(간격·구성 유지). **(4) 불필요 요소 제거** — 실제 내용에 없는 요소는 끈다. **(5) 세부 정리** — 컴포넌트로 못 덮는 부분만 커스텀해 다듬는다. **핵심: 큰 단위 먼저, 커스텀은 가장 늦게·좁게.** (피그마: 연결 유지하다 정말 안 되는 부분만 `detachInstance()` — Boolean 토글·visibility로 되면 detach 없이 처리 / 코드: 컴포넌트 prop으로 안 되는 부분만 커스텀 마크업)

   > **⚠️ '큰 단위 먼저'의 정확한 뜻 — 조립이지 복제가 아니다 (반복 마찰 1순위).** '큰 단위 먼저'는 **큰 컴포넌트/템플릿을 _인스턴스로 가져와 조립_**하라는 뜻이다. **기존 완성본(어제 작업물·다른 화면)을 통째로 복사(clone)해 고치는 것이 아니다.** 올바른 순서: ⑴ 가장 큰 DS 컴포넌트/템플릿을 **새로 인스턴스화/import** → ⑵ 그 slot·body에 다음으로 큰 단위(테이블 템플릿 등)를 **삽입** → ⑶ 불필요 요소를 끄고(visibility) → ⑷ 컴포넌트로 안 되는 세부만 커스텀(피그마=detach / 코드=커스텀 마크업). **완성본 복제는 토큰 바인딩·연결이 어긋나므로 금지.** (피그마 예: 모달 = `Modal` 컴포넌트 인스턴스 + `ModalBody` slot에 테이블 템플릿 조립. 코드 예: 아래 templates/*.md의 "모범 예제"처럼 기존 컴포넌트를 `import`해 조립 — `<table>`·`<input>`을 손으로 다시 짜지 않는다.)
5. **색 차이는 스타일을 직접 바꾸지 말고 컴포넌트 속성(state·color 등 variant)으로 표현** — 컴포넌트의 색 차이는 대개 둘 중 하나다: ① **상호작용 상태**(`state`: default / disabled / selected / hover / pressed) 또는 ② **분류·의미 색**(전용 variant 속성 — 예: Tag의 **`color`** = blue/red/gray, *구 `type`에서 개명*). 둘은 별개 속성이니 **상황에 맞는 속성**을 고르되, 어느 쪽이든 fill·텍스트 색을 **직접 수정(recolor)하지 말고** 그 **컴포넌트 속성**을 바꿔 표현한다. (피그마: `setProperties`로 `state`·`color` 등 변경, 변경이 어려우면 — 이미 detach됐거나 variant 없음 — 임의 채색 말고 기본값 유지 / 코드: `disabled`·`selected`·`variant`·`color` 등 **prop 사용**, `className`·인라인 스타일로 색 덮어쓰기 금지) 디자인 토큰·variant 정의를 우회한 수동 채색 금지. **피그마 패턴: 상태/색은 variant로 먼저 적용한 뒤, 텍스트까지 바꿔야 하면 그때 detach한다 — variant로 적용한 스타일이 detach 시 그대로 구워지므로 수동 채색 없이 텍스트만 편집 가능.**

15. **카피(문구) 띄어쓰기 규칙 (2026-07-03 지시)** — UI 카피(라벨·제목·버튼·플레이스홀더 등)의 띄어쓰기는 **한글 맞춤법 규칙**을 따른다. 복합어처럼 **붙여쓰기/띄어쓰기 둘 다 허용**되는 경우(예: 발신주소/발신 주소)는 **맥락상 이해하기 쉬운 쪽**을 택한다 — 한 단위 개념으로 읽히는 필드명·용어는 붙이고(예: "이메일 안내문 발신주소"), 구 단위로 읽히는 서술형은 띄운다(예: "안내 및 발표 명칭"). 같은 화면·시스템 안에서는 같은 용어의 표기를 통일한다.

### B. 코드 작성 전용

6. **새 페이지는 정해진 절차로** — `pages/XxxPage.jsx` 생성 → `pages/index.js` export → `App.jsx`의 `NAV_GROUPS`에 추가.
7. **컴포넌트는 완전 옵션화** — 모든 시각 옵션을 props로 노출, 기본값 명시, 인라인 스타일 금지.
8. **말줄임 텍스트는 hover 툴팁 — `TruncatingText` 사용 (구현 완료, 2026-07-03 '목표'→'사용' 승격)** — 좁아서 잘릴 수 있는 텍스트(목록·셀렉트 값·버튼 라벨 등)는 공용 `components/TruncatingText.jsx`로 감싼다(잘릴 때만 hover 시 전체 값 포털 툴팁). 이미 Button·Select·List·InlineFieldTrigger에 적용돼 있고 DateField는 동일 패턴 자체 구현. 새로 적용할 땐 드롭다운/overflow 컨테이너 안에서의 위치·잘림을 검증할 것.
9. **스크롤 영역은 `ScrollArea`** — 세로 스크롤이 생기는 곳은 네이티브 스크롤바를 쓰지 말고 `ScrollArea`(커스텀 오버레이 스크롤바)로 감싼다. `<ScrollArea maxHeight={…}>…</ScrollArea>`. **예외 — 네이티브 폼 컨트롤(`textarea` 등)은 내부 스크롤을 컨트롤 자신이 하므로 ScrollArea(div용 JS 오버레이)로 감쌀 수 없다. 그렇다고 네이티브 스크롤바를 CSS로 스타일링하면 hover가 브라우저 한계로 불안정(Firefox는 hover 불가)하고 auto-grow+ScrollArea로 감싸면 커서 추적이 깨진다 → 정답: 네이티브 스크롤(커서 추적)은 그대로 두고 `.hide-native-scroll`로 기본 바만 숨긴 뒤, ScrollArea와 같은 오버레이 thumb(scroll-bar 토큰, hover/드래그)를 그 네이티브 스크롤에 동기화해 얹는다(`TextArea.jsx` 참고).** **가로+세로 스크롤 중첩(2026-07-06 지시) — 세로 스크롤 영역이 가로 스크롤 콘텐츠 '안'에 중첩되면(표 바디 등) 세로 thumb를 그 안에서 그리지 말 것: 콘텐츠 전체 폭 우측(뷰포트 밖)에 붙어 가로 스크롤 위치에 따라 안 보인다. 정답: 안쪽은 thumb 없는 스크롤 요소(`.hide-native-scroll overflow-y-auto`)만 두고, 가로 스크롤을 안 받는 바깥 가로 ScrollArea가 `vScrollEl`로 그 요소에 연동해 세로 thumb를 그린다 — 세로 thumb는 항상 가로 스크롤 영향 없이 보여야 한다(`Table.jsx` 헤더 분리형 참고). thumb 규격은 시각 6px+히트 10px(`::before` 사방 2px 확장) 공통.**
10. **날짜·시간 표기는 통일** — 날짜+시간 `YY.MM.DD (HH:MM)` · 날짜만 `YY.MM.DD` · 범위 `A~B`(공백 없는 틸드) · 시작만 `A ~ 마감일 없음` · 마감만 `시작일 없음 ~ B`(모두 2자리 0 패딩). 직접 조립 말고 `src/utils/datetime.js`의 `formatDate`/`formatDateTime`/`formatDateTimeRange` 사용(상세는 foundation.md).

19. **반투명(알파) 라인은 border 금지 — outline/ring(바깥)으로 그린다 (2026-07-14 승격, 재발 3회)** — 알파 컬러 구분선·아웃라인을 CSS `border`(박스 안쪽)나 `ring-inset`으로 그리면 **요소 자신의 배경과 합성**돼 뒤 배경이 비치지 않는다(알파 의미 소실). 정답: `outline outline-1 outline-{토큰}` 또는 바깥 ring — 요소 밖에 그려져 뒤 배경과 합성된다. 판례: SegmentedTabs 선택 pill(ring-inset→outset)·조건 카드(border→outline)·ToolBar(border→outline). **예외: 드래그 고스트(setDragImage) 등 캡처되는 요소는 outline이 캡처 영역 밖이라 잘림 → border(안쪽) 유지.**

20. **순차 편집 UI의 키보드 Tab 이동은 체인 규약을 따른다 (2026-07-15 승격, 사용자 지시)** — 여러 편집 단계(트리거→팝오버→다음 트리거)가 이어지는 UI(스크리닝 조건 카드·수식 등)는 즉흥 keydown이 아니라 아래 규약대로 구현한다(공용 구현: `components/formulaFunctions.js`).
    - **⑴ 정거장 마커**: 체인에 참여하는 트리거/드롭 스트립에 `data-formula-chain` + `tabIndex=0` + `CHAIN_STOP_FOCUS` 링. **포커스 링은 `:focus` 기반**(programmatic `focus()`는 `:focus-visible` 미매칭). **체인 순서 = DOM 순서**이므로 마커 배치 순서가 곧 Tab 순서다.
    - **⑵ 트리거에서 Tab/Enter = 그 팝오버 열기** — 드롭·마운트 시 자동 오픈 금지(마우스 사용성 유지). 열 때 패널 포커스는 `setTimeout(0)` 지연(패널이 위치 계산 전 `visibility:hidden`이라 즉시 focus 실패).
    - **⑶ 열린 팝오버 내부 DOM을 Tab으로 순회** — focusable 수집 시 **`disabled` 요소는 반드시 제외**(`button:not([disabled])` 등 — 포함하면 실제 마지막 요소에서 경계 처리가 안 걸려 포커스가 패널 밖 GNB로 샌다, 2026-07-15 판례).
    - **⑷ 경계 규칙**: 마지막 요소에서 Tab → **완성이면 저장(onSave)+다음 단계 진행** / **미완성이면 onSkip = 저장 없이 닫고 다음 정거장으로**(값 미선택이어도 다음 카드/수식으로 진행). 처음 요소에서 Shift+Tab → 끝으로 순환.
    - **⑸ 다음 정거장 이동은 `focusNextChainStop(기준El)`** — 문서 전체 `[data-formula-chain]`에서 다음 마커를 찾아 포커스(setTimeout 0). 직접 `.focus()` 산발 호출 금지.
    새 순차 편집 UI를 만들 때도 이 규약(마커→트리거 오픈→내부 순회→경계 저장/skip→체인 이동)을 그대로 적용하고, 임의의 다른 Tab 순서를 만들지 않는다.

### C. Figma 작도 전용

11. **Figma 연결(Code Connect)은 개발환경 기능을 절대 훼손하지 않는다** — Figma 컴포넌트를 코드에 연결할 때, 개발환경에 구현된 컴포넌트의 기능·옵션(props·동작)에 **어떤 문제도 발생시키지 않아야** 한다. Code Connect는 코드↔디자인을 잇는 단방향 메타데이터 매핑일 뿐이므로, 연결을 위해 코드의 기능을 줄이거나 prop을 제거·변경하지 말 것. Figma 컴포넌트가 코드보다 옵션이 적어도(기본형만 있어도) 그대로 매핑하며, 부족한 표현은 코드 기본값/수동 지정으로 둔다(코드를 Figma에 맞춰 깎지 않는다). 매핑 후 코드 컴포넌트가 변경되지 않았는지 확인한다.
12. **에러 툴팁이 있는 입력은 감싸는 프레임의 clip content를 끈다** — Input 등의 에러 상태는 인풋 박스 아래로 absolute 오버레이(툴팁)로 표시된다. 이 인스턴스를 담는 프레임/오토레이아웃에 `clipsContent`(clip content)가 켜져 있으면 툴팁이 잘리므로, 에러 툴팁이 나오는 input을 Figma에 배치할 때는 그 상위 프레임들의 clip content를 **모두 꺼야** 한다. (코드 등가: 툴팁을 감싸는 요소의 `overflow: visible`로 잘림 방지)
13. **Figma에 테이블을 그릴 때는 연결된 table 컴포넌트를 detach해서 편집한다 — 컬럼·행 대량 편집이나 슬롯(ModalBody 등) 내부 편집 시 권장** — 연결된 `table` 컴포넌트(state 변형: table/outline/empty)를 **인스턴스로 생성 → `detachInstance()` → 셀/행을 자유롭게 편집**(컬럼 축소·행 정리·텍스트 입력)한다. detach해도 색·간격·구분선·텍스트 스타일 등 **디자인은 그대로 유지**되고, 잃는 것은 그 결과물의 컴포넌트 연결뿐이다. **단 detach 직후 자식들의 FILL 사이징이 풀릴 수 있다(2026-07-03 확인)** — detach하면 곧바로 프레임·테이블·바디 컨테이너·각 행의 `layoutSizingHorizontal='FILL'`을 일괄 재설정할 것(빼먹으면 원본 폭으로 남아 잘림/어긋남). **안정성(2026-06-24 검증):** table 컴포넌트 자체는 건강하다(단독 인스턴스는 `findAll`·셀 편집 모두 정상, 고스트 0). 다만 **`ModalBody` 같은 슬롯 안에 넣으면 슬롯 미러링 때문에 'node not found'류 노드(고스트)가 생겨** 깊은 순회·저장 참조 편집이 불안정하다 → **슬롯 안에서 표를 편집할 땐 detach가 안전**. 단순 인스턴스를 한두 군데만 손대는 정도면 detach 없이도 가능(과거의 '필수' → 현재 '대량/슬롯 편집 시 권장'). 단순·안정 컴포넌트(Button·Input·Tag 등)는 연결 인스턴스를 유지한다. **체크박스 셀: 테이블에 체크박스 셀을 쓸 때 셀 너비는 44px(FIXED)로 고정한다**(헤더·본문 동일, 콘텐츠 가운데 정렬). (코드에서는 detach 없이 DS `Table` 컴포넌트를 그대로 조립)
14. **DS 컴포넌트·템플릿은 Code Connect로 연결된 것을 찾아 인스턴스화한다** — Figma 작도에서 개별 컴포넌트(Button·Input·Select·Tag·Checkbox·Table·Pagination 등)**뿐 아니라 템플릿(TableTemplate 등 복합 단위)도 Code Connect로 연결돼 있다.** 그러므로 **레이어를 막 뒤지거나 임의로 새로 그리지 말고 Code Connect 맵에서 찾아 쓴다** — 규칙 3·4(페이지>템플릿>컴포넌트, **큰 단위 먼저**)에 따라 **템플릿으로 덮이면 템플릿을, 안 되는 부분만 개별 컴포넌트를** 찾아 조립한다. 이 파일의 DS 자산은 **컴포넌트·템플릿 전부 Code Connect 연결돼 있다(Editor도 2026-06-30 연결 — 2026-07-03 라이브 확인).** `get_code_connect_map(nodeId)`은 `{figmaNodeId: {componentName, source}}`를 주므로 이를 **뒤집어 "이름 → Figma nodeId"로 역조회** → `getNodeByIdAsync → createInstance`로 꽂는다(메모리 레시피에 node id가 있으면 그걸 우선 사용). **주의:** ① 맵은 **조회한 노드의 서브트리 범위만** 반환하므로 한 번 호출의 부분 결과만 보고 "미연결"이라 단정하지 말 것(필요하면 여러 노드/페이지로 확인). ② 연결 여부가 의심되면 추측 말고 `get_code_connect_map`으로 **라이브 확인**. ③ variant뿐 아니라 **컴포넌트/템플릿 SET 노드도 매핑돼 있어야** 읽기/Dev Mode 표시가 안정적. (상세·현황: 메모리 `reference_code_connect_coverage`)

## 작업 전 확인

- 작업 시작 전 해당 유형 파일을 먼저 읽고, 그 안의 **완료 체크리스트**를 작업 종료 시 **항목별로 출력하며 검증**한다(생각만으로 넘기지 말 것 — Figma ⓪체크의 코드판, 2026-07-06).
- 규칙에 없는 새 패턴이 필요하면 임의로 만들지 말고 사용자에게 확인한다.

## 규칙 거버넌스 (2026-07-06 — 규칙 비대화 방지)

- **기계 검사 우선**: 커밋 전 `npm run check:rules`(design-system/scripts/check-rules.mjs) 실행 — 임의 간격/행간(2b)·기본 스페이싱(2a)은 오류, HEX(1b)·카피 띄어쓰기(15)·모달 고정 maxHeight(18)는 후보로 자동 검출. **grep으로 잡을 수 있는 규칙은 문서를 늘리지 말고 이 스크립트에 추가한다.**
- **승격 기준**: 재발 2회 이상 또는 사용자 지시일 때만 번호 규칙로 승격. 1회성 판단은 원장 판례로만 남긴다.
- **가지치기**: 신설 규칙이 5개 쌓일 때마다 전체 감사(안 쓰는 규칙 통합·삭제, 비대 행 아카이브 이관)를 실행한다.
- **체크리스트 상한**: 템플릿당 핵심 10항목 이내 — 넘치면 통합.

## 작업 후 — 규칙 사용 기록 (`rule-usage.md`)

design-system 규칙을 **실제로 적용한 작업이 끝나면** `rule-usage.md`를 갱신한다(규칙 튜닝용 원장).

- 그 작업의 판단을 **실제로 좌우한 세부 규칙**만 적용 횟수 +1, 최근 적용일 갱신. 단순 언급은 세지 않는다.
- 규칙이 **모호했거나, 예외가 필요했거나, 다른 규칙과 충돌**했으면 해당 행 "마찰·애매 메모"에 날짜와 함께 한 줄 누적.
- 메모가 쌓인 규칙은 파일 하단 "튜닝 후보"로 승격해 추후 보완 대상으로 남긴다.
- **행이 비대해지면(수천 자) 판례 원문을 `rule-usage-archive.md`로 이관**하고 행에는 요약+`§키` 포인터만 남긴다(2026-07-03 1차 이관: 1a·2a·3c·4c·7a·11a·13a). 아카이브는 라우팅 대상이 아니며 상세 근거가 필요할 때만 연다.
- 자동 카운트는 불가(규칙은 도구 호출이 아님)하므로 이 갱신은 규율 기반 best-effort다.
